Status at the time of rendering determines the card variant shown: Accepted→still monitoring = monitoring card; Accepted→completed = successful card (green); Accepted→rolled back = reverted card (red); Declined = declined card (grey, opacity 0.55); Modified = successful card with amber "Modified" tag in change-block. Reverted cards: Red top bar (rt-reverted). Red border (card-reverted). Grey change block — shows original change + "(then reverted)". Outcome block (red) — "Auto-reverted by Radar" + accepted/reverted dates + KPI degradation reason. Footer meta only — no action buttons. Read-only label "⟳ Reverted".
5th tab decision: Reverted is a meaningfully different outcome from Successful — it means the change was harmful and rolled back. Folding it into Successful would be misleading. A 5th tab keeps outcomes clear. Tab order: Pending → Monitoring → Successful → Reverted → Declined. Reverted badge is red.
6. RADAR JOB — Background evaluation logic
1
Trigger — how often does Radar run?
Runs every 60 seconds via a background thread started in app.py at startup (threading.Thread(target=radar_loop, daemon=True).start()). Uses time.sleep(60) loop. No Celery/Redis required — simple enough for current scale.
2
Query — which recommendations does Radar evaluate?
SELECT * FROM recommendations WHERE status = 'monitoring' AND monitoring_ends_at <= NOW()
Also checks monitoring_minutes: if rule has monitoring_minutes set, Radar uses accepted_at + INTERVAL monitoring_minutes MINUTES as the end time instead of monitoring_ends_at.
3
Evaluate — KPI check per recommendation
For each eligible rec, Radar checks the relevant KPI from ro.analytics.campaign_features_daily:

Budget rules: Compare roas_w7 now vs at time of acceptance. If ROAS dropped ≥ 15% → revert.
Bid rules: Compare roas_w7 or cvr_w7 now vs at acceptance. If dropped ≥ 15% → revert.
Status rules: No KPI check — always → successful after monitoring period.
✓ KPI holds — mark Successful
UPDATE recommendations SET status='successful', resolved_at=NOW()
Log to changes table: action_type='radar_resolved', executed_by='radar'
✕ KPI degraded — Revert + mark Reverted
UPDATE recommendations SET status='reverted', resolved_at=NOW(), revert_reason='ROAS -18%'
Execute reverse change via execution engine (dry_run=False)
Log to changes table: executed_by='radar_revert'
4
File structure
New file: act_autopilot/radar.py — contains radar_loop() and evaluate_monitoring_rec(rec).
Imported and started in act_dashboard/app.py at startup.
Uses existing warehouse.duckdb connection (writable). Reads from ro.analytics.* for KPI data.
Scope note: Radar does NOT connect to the Google Ads API in Chat 29. The "reverse change" on revert is a database-only status update (sets recommendation back to original value). Live API execution is a future chat. This keeps Chat 29 scoped and testable.
